Data Import & Update Guide
==========================

This guide explains how to import data into RepoLens and how to run updates while the application is actively serving users.

The import process is responsible for two key tasks:

1.  **Configuration Sync**: Loading your project settings, component definitions, and team structures from your ``config.yaml`` into the configuration database.
2.  **Analysis Import**: Scanning your Git repository, calculating metrics (like complexity and churn), and populating the analysis database.

In RepoLens, all data extraction and updating is handled exclusively via Docker containers using the ``manage.sh`` script.

Initial Data Import
-------------------

To run the initial import for a repository, use the ``manage.sh`` script. This command mounts your local repository into the Docker container, runs the configuration sync, and then performs the full analysis.

**Command:**

.. code-block:: bash

   ./manage.sh import /absolute/path/to/your/repository

.. tip::
   Instead of passing the absolute path every time, you can create a symlink named ``repo`` in the RepoLens root directory pointing to your target repository:
   
   .. code-block:: bash
   
      ln -s /path/to/target/repo ./repo

   If you do this, you can simply run ``./manage.sh import ./repo``.

**Prerequisites:**

*   Docker must be running.
*   You must have a valid ``config.yaml`` in your project's ``config/`` folder.

Incremental Update Workflow (Zero Downtime)
-------------------------------------------

RepoLens is designed to support background updates without requiring you to shut down the frontend or the backend API.

The importer operates incrementally: it checks which commits are already present in the database and only analyzes new commits. To ensure this process is fast and doesn't lock the live database, RepoLens uses two separate database paths defined in your ``config.yaml``:

*   ``project.db_path``: The directory where the *live* databases used by the backend reside (e.g., ``/app/data``).
*   ``project.db_update_path``: The directory where the importer writes new data (e.g., ``/app/data/update_data``).

**The Update Process:**

1. **Prepare the Update Directory**
   For the incremental import to work efficiently, the importer needs to know the current state. Therefore, the live databases should be copied into the update directory before starting the import.

   *Note: If you are using the default configuration paths provided by RepoLens, this directory structure and file separation is handled automatically.*

2. **Run the Importer**
   Run the import command just like the initial import. The script will detect the existing databases in the ``db_update_path``, identify new commits in the repository, and append the new analysis data.

   .. code-block:: bash

      ./manage.sh import /absolute/path/to/your/repository

3. **Refresh the Live Database**
   Once the importer has finished running in the background and the databases in the ``db_update_path`` are fully updated, you need to swap them with the live databases.

   Use the ``refresh`` command to perform an automated swap. This will briefly pause the backend, back up the old databases, move the new ones from the staging area to the production area, and restart the backend.

   .. code-block:: bash

      ./manage.sh refresh

   *Note: This command requires the ``yq`` utility to be installed on your host machine to parse the configuration file.*

By following this workflow, your users can continue analyzing the existing data on the dashboard while the heavy lifting of parsing new commits happens transparently in the background.

Troubleshooting
---------------

*   **Database Locks**: If you attempt to update the live database directly without using the ``update_data`` directory, you may encounter SQLite/DuckDB file lock errors because the backend API is actively reading from it. Always use the separate update path.
